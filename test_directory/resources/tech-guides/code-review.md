# コードレビューガイドライン

## 目的

コードレビューはバグを見つけるだけでなく、知識共有・設計の改善・チームの品質基準維持のために行う。

## レビュアーとして

### 確認すること

1. **正確性**: バグや論理的な誤りがないか
2. **可読性**: 変数名・関数名が意図を表しているか
3. **セキュリティ**: 入力バリデーション、認証・認可、秘匿情報の扱い
4. **パフォーマンス**: N+1クエリ、不必要なAPI呼び出しなど
5. **テスト**: カバレッジが十分か、境界値テストがあるか

### コメントの書き方

コメントはカテゴリを明示する:

```
[blocking]  マージ前に修正必須
[nit]       小さな改善提案（マージ可）
[question]  疑問・確認
[praise]    良い実装の指摘
```

例:
```
[blocking] この関数はSQL文を直接結合しているためSQLインジェクションのリスクがあります。
           パラメータ化クエリを使用してください。

[nit] `isLoading` より `isSubmitting` の方が意図が明確かも。

[question] この処理は並列で実行できますか？Promise.all の方が速そうですが。
```

## PR作成者として

- レビュー依頼前にセルフレビューを実施する
- 変更の背景・意図をPR本文に記載する
- 大きなPRは事前に設計レビューを依頼する（PRは500行以内を目安）
- コメントへの返信は24時間以内に

## タイムライン

| アクション | 目標時間 |
|-----------|---------|
| レビュー開始 | 依頼から4時間以内（営業時間内） |
| レビュー完了 | 依頼から1営業日以内 |
| コメント対応 | 翌営業日中 |
